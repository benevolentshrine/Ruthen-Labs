# RuthenLabs Project Documentation

RuthenLabs is a sovereign, local-first AI development environment and infrastructure designed to replace cloud-dependent AI wrappers. It focuses on privacy, high performance, and absolute local control over AI agents and tools through a multi-layered security and orchestration architecture.

## 1. Core Architecture

The project is modularized into three primary components that communicate via high-performance Unix Domain Sockets (UDS). This "Clean Room" design ensures that AI-generated code never has direct, ungated access to the host system.

## 2. Component Deep Dive

### 2.1 Sandbox (Security Cage)
The **Sandbox** (formerly Boru) is the primary enforcement point for security. It operates under a **Zero-Trust** mandate, assuming all code generated by LLMs is potentially malicious.

- **Security Gate**: Implements a pre-execution gate that evaluates code against a policy matrix.
- **Resource Isolation**: Enforces memory and CPU limits to prevent "CPU-melt" or exhaustion attacks.
- **Filesystem Shadowing**: Uses a shadowing mechanism to protect critical files while allowing agents to work in isolated sessions.
- **Communication**: Listens on `/tmp/ruthenlabs/sandbox.sock` using JSON-RPC 2.0.

### 2.2 Indexer (Memory/Indexer)
The **Indexer** (formerly Yomi) provides the project's long-term and short-term memory. 

- **Fast Indexing**: Uses a parallel directory walker and content inspector to build searchable indices.
- **Semantic Search**: Enables the Orchestrator to inject relevant file context into LLM prompts via a `/search` command.
- **Ignore Logic**: Respects `.indexerignore` and `.gitignore` to keep sensitive data out of the index.
- **Communication**: Listens on `/tmp/ruthenlabs/indexer.sock`.

### 2.3 Orchestrator (Conducting Brain)
The **Orchestrator** (formerly Suji) is the central interface for the user, built using the Bubbletea TUI framework.

- **Agent Management**: Handles chat history, tool calling, and model switching (via Ollama).
- **Security Policy Sync**: Hardware-enforced policy synchronization with the Sandbox.
- **MCP Integration**: Supports the Model Context Protocol for connecting to external tool servers.
- **TUI Interface**: Provides a rich terminal experience for managing local AI workflows.

## 3. Communication Protocol

All components integrate through **Unix Domain Sockets (UDS)** using newline-delimited JSON (NDJSON) or JSON-RPC 2.0. This ensures:
- **Low Latency**: Near-instant communication between components.
- **Isolation**: No network ports are opened, mitigating remote attack vectors.
- **Strict Contracts**: Components interact through well-defined JSON schemas.

---

## 4. Security Principles

RuthenLabs is built on several key security pillars:
1. **Zero Trust**: No code is inherently safe; everything is sandboxed.
2. **Local First**: Data never leaves the machine unless explicitly piped to an external tool.
3. **Hardware Enforcement**: Critical policies are designed to be hardware-synced for maximum integrity.
4. **Context Blindness**: The Sandbox is context-blind, evaluating code only against its strict policy matrix.
